## 1. 概述与设计目标

Gulugulu 从"单只咕噜鸭桌宠"升级为**桌面精灵养成游戏**：一个透明无边框置顶小窗里的宠物小精灵/数码宝贝风格世界。产品定位是**编码伴侣 + 精灵养成**——你写代码，它吃 Token 成长；你休息，戳它两下打工赚钱；你睡觉，蛋在后台照孵。咕噜鸭不再是唯一主角，而是六只初始精灵中"一般属性"的那一只（唯一沿用现有 PNG 帧动画的精灵）。

设计支柱：

1. **贴合编码场景**（2026-07-21 改写）：**AI 干活喂经验、敲键盘回精力、你亲手点击赚金币**——AI 产出的 Token 折算成**主宠经验**（认真干活它就长），敲键盘与挂机只回精力（顺手敲字 / 开着就行）；**金币只从你亲手点击进入游戏**（见 §3.3/§5 与 `InteractionEconomy.md`）。
2. **窗口即世界**：所有玩法在透明窗口内完成，零网页依赖；窗口尺寸随内容自适应——宠物区固定置顶、水平中心锚定，高度向下生长。
3. **轻数值、强反馈**：每个数值都有水龙头和水槽，精力是点击收入的节流阀，账号级软上限兜底防通胀；每次操作都有即时动效（飘字、弹跳、裂壳）。
4. **可替换的占位美术**：MVP 全部用 SVG 组件占位，锚点与现有 PNG 帧系统对齐，后续可无缝换成 AI 生成形象。

## 2. 核心循环

玩家点击精灵打工，**同时获得金币与经验**（消耗精力；精力耗尽则精灵趴下充电、恢复到 10% 站起）；**点击是金币的唯一来源**，受每日全局 1000 次额度约束。经验另有一条水龙头：编码产生的 Token 折算成**主宠经验**（见 §3.3）。精力靠两条途径回复：挂机自然恢复、敲键盘（每个按键化作键帽飞向主宠）。金币在商店购买不同属性的蛋；蛋在孵化屋按真实时间孵化成精灵；两只**同阶且满级**的精灵拖到一起融合成一颗高阶蛋，回到孵化环节，产出更高阶精灵。孵化屋升级扩容同时孵化数，后院升级扩容持有上限，超容需放生（返还部分金币）。

### 3.3 等级：点击 + Token 双成长 + 两种精力恢复（2026-07-21 修订，权威定义见 `InteractionEconomy.md` §3/§4）

**金币只来自点击打工**：每击 `(1 + 等级) × 5^(阶−1)` 金币，与经验同发。**经验有两条水龙头**：① 点击打工每击 `4 × 5^(阶−1)` exp（1 阶 4 / 2 阶 20），受精力（每击 2 点）与每日全局 1000 次点击额度双重约束；② **AI 产出的 Token 折算成主宠经验**（见下文「Token→经验」）。挂机与敲键盘**不产出任何经验或金币，只回精力**：

| 恢复途径 | 换算（×5/阶） | 1 阶回满一管（200 点） |
|---|---|---|
| 挂机自然恢复（全员） | 3s/点 × 5^(阶−1)，离线同速 | 10 分钟 |
| 敲键盘（仅主宠） | 1 键/点 × 5^(阶−1)，全局钩子、防重复防宏 | 200 键 |

**Token→经验**（第二条经验水龙头）：AI 编码产出的 Token 按四分加权口径（cache_read/cache_create/output/input = 0.1/1/5/1）累计，每 `tokensPerExp=40` 加权单位折 1 点**主宠经验**（全阶统一、不乘 tierFactor——经验体系本身已按阶放大）；账本增量结算、余数存 `pet.tokenBuffer`（语义为经验进度）。**主宠满级或缺席时 Token 整段浪费，绝不溢给其他宠**；Token 喂养不产金币、不占每日点击额度。

**精力喂养去向**（保证进食动画永远对应可见收益）：键盘充能**只喂主宠、无溢出转移**——主宠满管即丢弃（轻提示"精力已满"，不折金币）；挂机各宠独立结算。（Token 已移出精力体系，改喂主宠经验，见上。）

**等级曲线**（权威见 `EconomyScaling.md` §2.1）：Lv n → n+1 所需经验 = `4 × n`。1 阶满级 **Lv10**（累计 180 exp = 45 击）；2 阶满级 **Lv20**，所需 = `10 × n`（累计 1900 exp = 95 击）。满级精灵卡片显示星标，是融合的前提。

**配比意图**：1 阶满级 45 击 ≈ 半管精力（满管 100 击），开局几分钟亲手点满第一只；2 阶满级 95 击 ≈ 一管精力，是次日主线。编码时键帽雨持续回精力、AI 的 Token 持续喂主宠经验，休息时一轮点击把攒下的精力变成金币与成长——"AI 干活喂经验、敲键盘回精力、你亲手点击赚金币"。

**历史存量并轨**：存档首次创建时读取一次 `gulugulu-progress.json`，将**所有项目的 experience 求和**，一次性折算为 `min(总和, 200)` 金币的开局礼包；之后游戏只消费账本的增量——不改后端换算，不重算历史，字节偏移防重复机制原样保留。

## 4. 蛋与孵化系统

### 4.1 蛋的阶级

- **1 阶蛋**：商店购买，六种属性各一种，孵出对应初始精灵。
- **融合蛋（2~6 阶）**：不出售，仅由融合产出，孵出 `speciesByRecipe` 配方对应的精灵（融合 2.0，见 `FusionSystem.md`）。
- **教学蛋**：开局免费预置在孵化槽的一般蛋，60 秒速成，保证 1 分钟内见到第一只精灵。

### 4.2 孵化时长（真实时间，离线照孵）

| 蛋 | 时长 |
|---|---|
| 教学蛋 | 60 秒 |
| 一般蛋 | 3 分钟 |
| 火 / 水 / 草蛋 | 5 分钟 |
| 电 / 冰蛋 | 8 分钟 |
| 2 阶蛋 | 30 分钟 |

孵化存绝对时间戳 `hatchAt`，关机照常孵，重启即时结算，不可加速（加速道具留待后续版本）。孵化完成的槽发光抖动，点击播裂壳动画收取；**后院满员时精灵留在槽内等待收取（占槽），提示先去后院放生腾位**。买蛋或融合产蛋时若无空槽，蛋进入库存，可稍后手动入槽（`place_egg`）。

## 5. 打工与精力系统

（v1.1 重构，权威定义见 `InteractionEconomy.md` §3/§4/§5）

- **打工入口有两个**：① `uiMode = menu`（菜单已展开）时点击宠物本体 = 对主宠打工；② 后院界面直接点精灵 = 对该精灵打工（弹跳 + 金币飘字 + 精力条即时扣减），无需来回切换主宠。菜单未展开时点宠物是"摸头 + 弹菜单"（见 §10.2 的完整消歧规则）。
- **点击收益**：`(1 + 等级) × 5^(阶数−1)` 金币/次 + `4 × 5^(阶数−1)` exp/次。1 阶 Lv1 = 2 金，Lv10 = 11 金；2 阶 Lv1 = 10 金，Lv20 = 105 金。满级精灵点击不再产经验，金币照发、照计额度。
- **每日额度**（防通胀总闸，替代 v1.0 的点击金币软上限）：账号级每日 **1000 次**有效点击（消耗精力即计数），跨宠共享由玩家分配；用尽后点击进入**纯抚摸模式**——不耗精力、无产出、只播爱心特效与温馨气泡，不禁点不报错。
- **精力 = 点击次数**：每只精灵独立，上限 **200 点**，每次点击消耗 **2 点**（满管恰好 100 击）。
- **自然恢复**：1 阶 1 点/3 秒（10 分钟回满），每升一阶恢复耗时 ×5（2 阶 50 分钟）；清醒、睡眠、离线同速，用 `staminaUpdatedAt` 时间戳惰性结算，无 tick 无离线补偿逻辑；**全员生效**。另有一条主动恢复途径（敲键盘，仅喂主宠、无溢出转移）见 §3.3。
- **精力耗尽与 10% 唤醒**：打到 0 进入 `exhausted` 恢复期——趴下充电（lie 姿态 + 💤 + 恢复进度），**仅打工点击无效**（温柔驳回演出，非报错）；菜单、商店、孵化屋照常可用。恢复到 **≥10%（20 点）站起、恢复接受点击**（伸懒腰唤醒演出）；醒着时精力 200→1 全程站立可点。恢复期间挂机与键盘照回精力，喂到 ≥20 直接喂醒；此时 Token 只喂经验、不参与唤醒。

### 7.1 AI 融合（2026-07 上线）

融合升级为**双轨制**，由本地编码代理 CLI 现场创造新物种：

- **CLI 前置（2.0 修订）**：点融合预检本地 `claude --version`（优先）→ `codex --version`（兜底）。CLI 不可用时**不再拒绝融合**——已解锁的 AI 变种照常复用、仅去掉"解锁新变种"这一路（`FusionRecipeSlots.md` §3.4），融合台显示"未检测到 CLI，本次在已解锁物种中抽取"信息条。生成用用户已登录的 CLI 账号，不引入任何 API Key。
- **掷骰（2.0 修订 → 槽位阶梯）**：**同物种融合永不掷骰**（确定性升阶）；异物种配对走**配方 11 槽位阶梯**（`FusionRecipeSlots.md` §3）——触发 AI 的总概率按配方元素数缩放（2/3/4/5/6 元素 = 60/40/20/10/5%），0 号固定物种 = `1−A(e)`；1~10 号 AI 变种渐进解锁（小序号未集齐不出大序号、2 号+ 需先集齐 0 号），掷中已解锁槽**复用**同一物种、掷中前沿新槽才 **AI 生成**。~~旧：单次 `aiFusionChance`(0.5) 掷骰、无上限唯一物种~~。
- **先提交后生成**：AI 路径立即消耗双亲 + 扣费，产出一颗**挂起融合蛋**（species 先写**并集对应的图鉴物种**作兜底）按结果阶时长孵化；后台 worker 并行调 CLI 生成（每 provider 初次 + 带校验错误的纠错重试各一次，单次超时 5 分钟）。成功 → 注册自定义物种进存档（`GameSave.customSpecies`）并改写蛋的 species；**孵化完成时仍未生成完 → 孵出并集对应的图鉴物种**（2.0 起不再兜底咕噜鸭）。应用中途退出不丢进度：重启后 worker 重扫挂起蛋续跑。

## 8. 后院与容量

| 等级 | 持有上限 | 升级费 |
|---|---|---|
| Lv1 | 3 只 | 初始 |
| Lv2 | 5 只 | 300 金 |
| Lv3 | 8 只 | 1200 金 |

- **放生**：后院选中精灵 → 放生（二次确认）→ 返还 `⌊对应蛋价 × 5%⌋ + 等级 × 5` 金币（**向下取整**；满级 1 阶一般 = 8 + 50 = 58 金 < 蛋价 160，满级电/冰 = 15 + 50 = 65 金 < 300——"刷蛋练满卖掉"永远不划算，防通胀）。**高阶精灵的"对应蛋价"取等效蛋价**（乘法口径，见 `EconomyScaling.md` §8）= `(Σ 物种各元素基价) × eggTierPriceMultiplier^(阶数−1)`（例：2 阶 {火,冰} = (240+300) × 20 = 10800）。
- **防死局**：最后一只精灵禁止放生（融合例外，见 §7）；精力自然恢复保证点击永远能重新开张，点击收益保证收入永不归零。
- **防通胀**（2026-07-21 修订）：金币水龙头唯一——**只有点击才产金币**，键盘/挂机只产精力（自封顶）、Token 只产**主宠经验**（不产金币、不占额度）；账号级**每日 1000 次点击额度**是金币的唯一总闸。经验虽多一条 Token 水龙头，但**经验不是货币**——它只把主宠推向满级（融合前提），加速的是融合节奏而非金币供给，故不破坏通胀模型；且 Token 经验只入主宠、满级即废，天然自封顶。保留等级加成后 2 阶满级理论日收入（金币）上限 ~10.5 万金；**分阶指数水槽（蛋价/融合费/设施升级）已由 `EconomyScaling.md` v1.2 调价 pass 落地**匹配（原 v1.1 的物价缺口已闭合）；更高阶水槽：后院 3→50、孵化屋 8 槽、6 阶融合费。

## 12. 技术实现要点

### 12.1 存档（`save_manager.rs` + `gulugulu-save.json`）

与 `gulugulu-progress.json` **分文件、单向依赖**：progress 仍是 token 事实账本（不动），save 只消费其增量。**仿 `SharedCodexState` 的 `Arc<Mutex<GameSave>>` + `.manage()` 常驻内存模式**（注意：不是 ProgressStore 的每次整读整写磁盘模式），每个写命令成功后整写 JSON 落盘。`#[serde(rename_all="camelCase")]`，`types.ts` 全量镜像（Rust↔TS 类型对等是仓库硬性规则）：

```rust
struct GameSave {
  version: u32,                    // =3（v1.1 交互经济重构；v2→v3 迁移见 InteractionEconomy.md §9）
  coins: u64,
  pets: Vec<PetInstance>,          // id, species, element, tier, level, exp,
                                   // stamina, stamina_updated_at,
                                   // key_buffer（键盘→精力余数）, token_buffer（Token→经验进度余数）
  eggs: Vec<EggInstance>,          // id, species, element, tier,
                                   // slot: Option<u8>, hatch_at: Option<i64> // None=库存
  hatchery_level: u8,
  yard_level: u8,
  active_pet_id: Option<String>,
  last_seen_project_tokens: BTreeMap<String, u64>, // Token→经验增量结算锚点，原始 token 口径（§12.4）
  daily: DailyCounters,            // date, clicks（日额度计数）；snack_stamina 已随漫游零食移除（2026-07-21，serde default 兼容，不升版本）
  tutorial_step: u8,
  last_seen_at: i64,               // 预留给 v2 离线挂机
}
```

所有 §9 常量集中在 Rust 侧 `game_config.rs` 的 `GameConfig`（含 `fusion_table`，键为排序后的 `"electric+fire"`；含日上限与软上限阈值），前端经 `get_game_config` 拉取，禁止两侧硬编码复制。

### 12.3 新增 IPC 命令（注册进 `generate_handler!`，共 12 个）

所有校验（钱、槽位、同阶满级、后院容量、日上限）在 Rust 做；写命令返回**完整新 `GameSave`**，前端整体 setState，避免双侧状态漂移：

### 12.4 Token 接入点（与 codex_adapter 的关系，2026-07-21 改为经验口径）

**口径：按 progress 账本的累计四分加权 token 增量喂经验，不用逐事件 delta。** `GameSave` 按项目记录 `last_seen_project_tokens`，`codex_adapter.rs::apply_agent_token_event()` 结算完 progress 后，用增量差值调 `game::feed_from_project_tokens(diff)`（内部按 `tokensPerExp=40` 折算喂**主宠经验**、余数入 `tokenBuffer`，**只入主宠、无溢出、只产经验不产金币，主宠满级或缺席则整段浪费**），并把 `fedExp`/`fedLeveledUp`/`fedPetId` 附在现有 `codex://activity` 事件负载里，同时以 `TokenFeedOutcome{petId,expGained,leveledUp,levelAfter,expAfter,wasted,fedBreakdown}` 为载荷推送新事件 `game://exp`；前端播 Token 芯片飞行 + fed 咀嚼 + `+N ✨经验` 飘字（满级则「Token 就当零嘴」提示）。账本自愈：progress 总数小于锚点（用户删了 progress 文件）时重置锚点视为 0 增量。历史存量折币礼包（首建存档一次性 ≤200 金）保留不变。

> **实现拆分（2026-07-21）**：旧 `logic_feed_energy` 已拆为 `logic_feed_keys`（键盘→精力，仅主宠、无溢出）与 `logic_feed_tokens`（Token→经验，仅主宠），`EnergySource` 枚举删除；`game://stamina` 的 `source` 恒为 `"keys"`。成就侧 `stats.total_tokens_fed` 仍按四分加权喂养单位累计（阈值 1M/50M/1B 不变），夜猫子旗标两条喂养路径都保留；`stats.first_maxlevel_done` 在点击与 Token 喂养两条路径都置位（Token 喂满级也能拿「首次满级」）。

### 12.5 状态机扩展（与 petEvents 的关系）

- 顶层 `uiMode: 'pet'|'menu'|'shop'|'hatchery'|'yard'|'fusion'|'status'`（7 个），独立于 `PetState`。`uiMode !== 'pet'` 时短路漫游 useEffect、暂停睡眠计时器与 `startNextFedIfReady` 出队；`petState` 机器语义不改。
- `PetState` 新增 `laboring`（点击打工瞬态 800ms，SVG 弹跳/guluduck 复用 pet_head）与 `exhausted`（映射现有 sleep 动画）。现有 `working` 语义是 agent 活动，**不复用**。
- **点击消歧**（与 §10.2 一致）：`handlePointerUp` 的点击分支按 `uiMode` 分发——`pet` 态派发 `user_click`（摸头 + 展开菜单条，原 `showStatusPanel` 逻辑并入状态页）；`menu` 态派发 `user_work_click → laboring`。
- **新增 PetState 的完整改动面**（不止两行映射）：`types.ts` 的 `PetState`/`PetEventType`、`petEvents.ts` 的 `stateAnimationMap`（`Record<PetState, AnimationKey>` 全量映射，缺项编译失败）与可选 `transientStateDurationMs`、`App.tsx` 的 `speechContextTags`（同为全量 Record）都要补齐；并把 `exhausted` 加入两个现有定时器的排除名单——"非 idle 状态 5 秒打回 idle"的 transient 超时 effect 和 `SLEEP_TIMEOUT_MS` 的入睡 effect（否则 exhausted 会被错误转成 sleeping），唤醒改由精力结算 ≥60 驱动。

### 12.6 窗口动态尺寸

**现状勘误**：现有右键菜单从前端 JS 调 `getCurrentWindow().setSize()`，但 `capabilities/default.json` 只授予 `core:default`、`allow-outer-position`、`allow-set-position`、`allow-start-dragging`——**缺 `core:window:allow-set-size`，该调用实际一直被 ACL 拒绝**，且被 `.catch(() => undefined)` 静默吞掉，并非已验证路径。

**方案**：动态尺寸统一走 Rust 命令 `resize_game_window(w, h)`（后端 `window.set_size()` 不经前端 ACL，无需改 capabilities；顺带把右键菜单的窗口扩展迁到同一命令）。前端实现 `useWindowAutoSize` hook：新增一个**由内容撑开高度的测量根节点**供 `ResizeObserver` 观察（不能观察 `.pet-shell`——它是 `100vw/100vh` 的视口盒，尺寸永远等于窗口；同时移除 `baseWindowSize` 内联钉死），120ms 防抖后调命令，minWidth/minHeight 钳制在 Rust 侧处理。

**锚定策略（已定）**：**顶边固定 + 水平中心固定**。高度变化只向下生长，无需位置补偿（Tauri 的 set_size 保持左上角不动、向下扩展，天然吻合）；宽度变化（260↔300↔320↔340）时在 `resize_game_window` 命令内由 Rust **配对调用 `set_position` 将 x 平移 −Δw/2**，保持窗口水平中心与宠物屏幕位置不动（set_size 与 set_position 在同一命令内先后执行，避免前端两次往返造成的抖动）。配套布局改动：`.pet-shell` 现有 `grid-template-rows: auto 1fr auto` 中宠物所在的 `1fr` 行改为固定高度 320px（否则窗口变高时宠物随 `1fr` 拉伸下移）。

**实现首日检查点**：`tauri.conf.json` 当前 `resizable: false`，Tauri 2 下程序化 `set_size` 一般不受其影响，但这是方案成立的前提，须第一时间实测；若受限，后备方案是配置改 `resizable: true`（无边框窗口无拖拽边框，用户仍无法手动缩放）。

`isTauri()` 为 false（浏览器预览）时 hook 空转，详见 §13 对预览模式的验收降级。

### 12.7 时间结算（v1.1 收拢为单一机制 + 刷新线程）

- **孵化与精力：绝对时间戳，无 tick**。孵化存 `hatchAt`，`now >= hatchAt` 即可收取，关机照常孵；精力按 `staminaUpdatedAt` 惰性计算 `min(200, stamina + elapsed / (3s × 5^(阶−1)))`，写操作前先结算再扣；时间戳晚于当前时刻时钳回（时钟回拨防呆）。
- **60s tick 线程只做刷新**：不再产出任何经验/金币（挂机经验与挂机产金已移除），每 tick 仅执行精力结算 + 日期翻转检查，并推送 `game://state` 让 UI 的精力条在挂机时也保持走动。键盘充能走独立的 `key_watcher` 节拍（250ms 特效批 / 1s 入账），见 `InteractionEconomy.md` §9。

### 12.8 与 avatar_manager 的关系

内置 guluduck manifest 与自定义头像**安装**流程不动；六只一阶与全部配方物种（融合 2.0 后共 63 只，见 `SpeciesMatrix.md`）作为 `renderer: 'svg'` 的内置条目注册。**右键菜单中打开 avatar-gen 网页的入口在 MVP 移除**（§10.1）；avatar-gen 服务与 `install_avatar_from_url` 能力保留在仓库，v2 用它为融合体动态生成形象。
